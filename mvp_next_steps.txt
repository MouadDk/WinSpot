  ✅ COMPLETED: QR CODE VISIT VERIFICATION SYSTEM
══════════════════════════════════════════════════════

  A full O2O (Online-to-Offline) verification loop
  was implemented. Here is exactly how it works:

  ┌─────────────────────────────────────────────────┐
  │  HOW THE QR VISIT FLOW WORKS (STEP BY STEP)    │
  └─────────────────────────────────────────────────┘

  1. MERCHANT GENERATES QR CODE
     - Merchant goes to their dashboard → "QR Visit
       Verification" section.
     - Selects one of their active offers from a
       dropdown.
     - Clicks "Generate QR Code".
     - The backend creates a QRVisit document in
       MongoDB with:
         • A unique UUID token (one-time use)
         • offerId, merchantId, label, winCoins
         • expiresAt = now + 1 hour
     - A QR code SVG is rendered on screen encoding
       the URL: /scan-qr?token=<uuid>
     - Merchant can also copy the raw token for
       manual sharing.

  2. INFLUENCER SCANS QR AT VENUE
     - Option A (phone camera): Scans the QR → opens
       the /scan-qr?token=... URL in their browser →
       the token auto-fills in the input field.
     - Option B (manual): Merchant gives/texts the
       token → influencer pastes it in the "QR Visit
       Check-In" panel on their dashboard.
     - Influencer clicks "Verify".

  3. BACKEND VALIDATES & CREATES PENDING TRANSACTION
     - Checks: token exists, not expired, not already
       scanned, influencer ≠ merchant.
     - Creates a Transaction document:
         • userId = influencer's MongoDB _id
         • type = "gain"
         • amount = winCoins from the QR
         • status = "pending"  ← KEY: not completed!
         • description = "QR Visit verified @ <venue>"
         • offerId = linked offer
     - Updates the QRVisit document:
         • scanned = true
         • scannedBy = influencer's _id
         • scannedAt = now
         • transactionId = the new Transaction _id

  4. BOTH SIDES SEE THE RESULT IMMEDIATELY

     MERCHANT DASHBOARD ("Recent QR Activity" table):
     ┌────────┬───────┬─────────────────┬─────────────┐
     │ Offer  │ Coins │ Scanned by      │ Transaction │
     ├────────┼───────┼─────────────────┼─────────────┤
     │ Café X │ 🪙 1  │ ✅ John Doe     │ ⏳ pending  │
     │        │       │    14:32        │             │
     └────────┴───────┴─────────────────┴─────────────┘
     - Shows influencer's first + last name
     - Shows scan time
     - Shows real transaction status from DB

     INFLUENCER DASHBOARD ("My Visit History"):
     - Shows venue name, scan timestamp, coin amount
     - Shows "pending" badge
     - Shows: "🤖 Pending transactions will be
       verified and approved by AI automatically."

  5. AI VERIFICATION (FUTURE — see Next Steps below)
     - An AI service will review the pending
       transaction, verify the visit evidence, and
       flip status from "pending" → "completed".
     - Only THEN will the WinCoins be credited to
       the influencer's balance.

  │  SECURITY & CONSTRAINTS                         │
  └─────────────────────────────────────────────────┘

  • QR codes expire after 1 HOUR
  • Merchant cannot scan their own QR code
  • Transactions are created as "pending" — no
    WinCoins are credited until AI approves
  • All actions are logged via Winston (activity.log)

  ┌─────────────────────────────────────────────────┐
  │ STEP 1: AI VERIFICATION SERVICE                 │
  │ Priority: 🔴 CRITICAL                           │
  └─────────────────────────────────────────────────┘

  The core reason transactions are "pending" — we
  need an AI to verify and approve them.

  IDEAS:
  a) Simple Rule-Based Verifier (MVP):
     - Cron job or webhook that auto-approves all
       pending QR transactions after a configurable
       delay (e.g., 15 min). This is a placeholder
       until real AI is ready.
     - Admin can manually approve/reject from the
       admin dashboard.

  b) Image-Based Proof of Visit:
     - Influencer uploads a photo/receipt at the
       venue after scanning QR.
     - AI (OpenAI Vision / Google Cloud Vision) 
       checks if the photo matches the venue or
       contains a valid receipt.

  c) Social Post Verification:
     - Influencer submits their Instagram/TikTok
       post URL.
     - AI scrapes the post metadata (location tag,
       hashtags, @mention) to verify it matches the
       merchant's offer.

  d) Geofencing Verification:
     - Compare influencer's GPS coordinates at scan
       time vs. the offer's location coordinates.
     - Auto-approve if within 200m radius.

  IMPLEMENTATION:
  - Create POST /api/qr/verify endpoint
  - Add "completed" / "failed" status transitions
  - Credit influencer balance on "completed"
  - Deduct from offer budget on "completed"
  - Notify both sides (WebSocket or polling)
